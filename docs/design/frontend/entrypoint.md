# Entrypoint — Condition List

## Goal

Give users a simple starting point to find and navigate to a condition they want to learn about, scoped to the current [space](spaces.md).

## Expected Behavior

- A page shows the conditions in the current space as a flat list
- The page header shows the space name and a switcher to jump to another space
- A search box at the top filters the list in real time by condition name or alias
- Clicking a condition navigates to its own condition page
- Each row shows a "This might be relevant if…" line written in plain patient language pointing at the specific distinguishing pattern that separates this condition from the others — not generic symptoms
- If the space defines probability data, the list is sorted by probability descending and each row shows a 1–5 dot indicator in a "Probability" column (see [probability.md](probability.md))

## Output Format

Routes:

- `/` — the **main** space (every condition).
- `/space/:id` — any other space defined in `data/spaces.yaml` (e.g. `/space/ibs`).

See [general.md](general.md) for the shared tech stack and visual design decisions, and [spaces.md](spaces.md) for how membership is resolved.

## Data Source

Two static files:

1. The global condition index at `data/generated/index.json`, generated by the CLI alongside condition data. Shape:

   ```json
   [
     {
       "id": "histamine",
       "name": "Histamine intolerance",
       "aliases": ["HIT"],
       "group_id": "immune",
       "group_label": "Immune & chemical sensitivity",
       "has_data": true,
       "relevance_signal": "Your gut symptoms appear within 30 minutes of red wine or aged cheese — not after most other foods."
     },
     ...
   ]
   ```

2. The spaces config at `data/spaces.yaml`. The frontend resolves the current space's membership (`include` + `exclude`) against the global index to produce the list shown on this page.

The SPA fetches both at startup. No backend server required.

## Constraints / Edge Cases

- If `index.json` does not exist, show a friendly empty state with instructions to run `cn generate`
- If `spaces.yaml` is missing, fall back to an implicit `main` space containing every condition
- If the current space resolves to zero conditions, show a friendly empty state explaining the space is empty
- Search should match aliases too (e.g. searching "HIT" finds Histamine intolerance)
- Search is scoped to the current space — it does not surface conditions outside the active space
- If `relevance_signal` is empty for a row (data not yet generated), the column shows an em-dash so the row layout stays aligned
- Subtypes (conditions with a parent) are included in the list and searchable, but could optionally be grouped or indented under their parent — leave this as a future enhancement

## Implementation

- Fetch `data/generated/index.json` and `data/spaces.yaml` (served via Vite's publicDir) at startup
- Resolve the active space (from the route) and filter the index to that space's members
- Pass the filtered list and the space's optional `probability` reference to `ConditionList`
