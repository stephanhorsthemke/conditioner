# Starting Points

## Goal

Give users a symptom-first way into the condition graph. A **starting point** is a
curated thematic landing page (e.g. IBS, brain fog) that presents the underlying
conditions worth ruling in or out — best-effort ranked by how often each shows up —
so someone who only knows their symptom can navigate toward candidate causes without
already knowing condition names.

Starting points replace the earlier "spaces" concept: instead of a generic
filtered view over the whole graph, each starting point is an owner-curated,
symptom-oriented entry with its own ranking and relevance signals.

## Core idea

A starting point is **not** a condition. It has no biology, symptoms, testing, or
management sections of its own. Its only job is to present a curated table of candidate
sub-conditions, optionally filterable by symptom patterns, with a plain-language
"Relevant if…" signal for each row.

Two kinds:

- `syndrome` — the starting point is also a diagnosable condition (e.g. IBS).
  `condition_id` links to its full condition page ("Learn about IBS the condition →").
- `symptom` — the starting point is a symptom people search for (e.g. brain fog).
  There is no linked condition page.

## Expected Behavior

- Each starting point has its own page at `/start/:id`, reachable from the
  [home page](entrypoint.md) cards and from [search](search.md).
- The page shows the starting point's label, an intro in informational framing, and
  (for `syndrome` kind) a link to the underlying condition page.
- Below the intro, a **table of candidate sub-conditions** is shown, ranked by
  probability descending (see [probability.md](probability.md)), each row linking to
  its condition page.
- Each row shows a plain-language **relevance signal** ("Relevant if…") describing the
  distinguishing pattern that points at that condition — generated by AI, not reviewed
  by a professional, and clearly labelled as such.
- If the starting point defines **symptom patterns**, a chip filter above the table lets
  the user narrow the list to conditions matching a pattern (e.g. "IBS-D",
  "Worse when you stand up"). Omitting patterns yields a plain table.

## Output Format

Route: `/start/:id`. See [general.md](general.md) for the shared tech stack and visual
design decisions.

## Data Source

`data/starting_points.yaml` — owner-edited. Each entry has:

- `id`, `label`, optional `full_label`
- `kind`: `syndrome` | `symptom`
- `condition_id`: for `syndrome`, the linked condition (null/absent for `symptom`)
- `intro`: plain-language introduction
- `symptom_patterns`: optional list of `{ id, label, description }` powering the chip filter
- `sub_conditions`: list of `{ id, probability, patterns }` — the candidate conditions,
  their 1–5 probability, and which symptom patterns each matches

The CLI generates two artefacts consumed by the frontend:

- `data/generated/starting_points/index.json` — one merged entry per starting point
  (labels, intro, sub-conditions, patterns, `has_signals`).
- `data/generated/starting_points/<id>.json` — the AI-generated relevance signals
  (`cn generate-signals`), loaded only when `has_signals` is true.

Sub-conditions reference IDs from the global condition index
(`data/generated/index.json`); rows whose ID is not in the index are skipped.

## Constraints / Edge Cases

- Relevance signals and rankings are AI-generated — the page shows an "AI-generated"
  label and a note that they are a research starting point, not a diagnosis
  (see [legal.md](legal.md)).
- A starting point with no resolvable sub-conditions shows a friendly "no sub-conditions
  yet" message rather than an error.
- If signals have not been generated (`has_signals: false`), the table renders without
  the signal text — the ranking still shows.
- Language stays informational throughout ("these are worth ruling in or out", not
  "you have X"), and steers toward professional help for anything serious.

## Implementation

- `data/starting_points.yaml` — owner-edited config.
- `web/src/types.ts` — `StartingPointIndex`, `StartingPointData`, `SymptomPattern`,
  `SubConditionRef` types.
- `web/src/pages/StartingPointPage.tsx` — fetches the index + per-id signals, renders
  intro and the sub-condition table.
- `web/src/components/SubConditionTable.tsx` — the ranked table with probability dots,
  relevance signals, and the symptom-pattern chip filter.
- `web/src/App.tsx` — the `/start/:id` route.
